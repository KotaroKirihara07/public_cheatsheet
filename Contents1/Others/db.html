#設計の分類と順番

  1. 論理設計

    2. 正規化やER図を用いたモデル設計。概念スキーマの設計。

  2. 物理設計

    3. サーバやストレージ等の物理的なハードウェアレベルの設計。内部スキーマの設計。

  3. 実装設計

    4. 特定のデータベース製品における具体的な構築の手順の決定。

#3層スキーマ

データの独立性を保証するための仕組み。内部スキーマと外部スキーマが互いに直接影響を及ぼしあわないようするために概念スキーマが存在する。

  • 外部スキーマ

    • システム利用者から見える機能等を定義するスキーマ。ビューが対象。

  • 概念スキーマ

    • データベースに保持するデータの要素やデータ同士の関係を記述するスキーマ。テーブルが対象。

  • 内部スキーマ

    • DBMSにどのように格納するか定義するスキーマ。ファイルが対象。

論理設計

概念スキーマを定義する設計。

  1. エンティティの抽出

    2. エンティティ (実体)を抽出する。つまり、テーブル名となり得るものを挙げる。

  2. エンティティの定義

    3. エンティティの属性や主キーなどを定義する。

  3. 正規化

    4. エンティティのフォーマットを整理する。無損失分解である第3正規形に到達すれば十分。

    正規形 状態
    第1正規形 一つのセルに一つの値のみ含む状態
    第2正規形 主キーの一部から非キー属性への関数従属(部分関数従属)が解消された状態
    第3正規形 非キー属性から非キー属性への関数従属(推移的関数従属)が解消された状態

物理設計

内部スキーマを定義する設計。

  1. テーブル定義

    2. 論理設計のアウトプットをもとにテーブルを定義する。

  2. インデックス定義

  3. ハードウェアのサイジング

    4. 必要なストレージの容量、ストレージのディスクI/O性能、CPUの性能、メモリの性能を決定する。

  4. ストレージの冗長構成決定

    5. 求められる性能と信頼性から、RAIDのレベルとハードディスクの数を決定する。

  5. ファイルの物理配置決定

    6. データファイル、インデックスファイル、システムファイル、一時ファイル、ログファイルをどのように配置するか決定する。
