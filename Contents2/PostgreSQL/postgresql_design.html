データベースの設計

2024/05/19

このページではデータベースの設計手法についての説明を行う。

基本

#設計の分類と順番

  1. 論理設計

    2. 正規化やER図を用いたモデル設計。概念スキーマの設計。

  2. 物理設計

    3. サーバやストレージ等の物理的なハードウェアレベルの設計。内部スキーマの設計。

  3. 実装設計

    4. 特定のデータベース製品における具体的な構築の手順の決定。

#DOAとPOA

DOA (Data Oriented Approach)とはデータの設計をした後にプログラムを作成する考え方。 POA (Process Oriented Approach)とはプログラムを書いた後にデータ設計をするという考え方。 基本的にはDOAで開発を行ったほうが効率が良い場合が多い。

#3層スキーマ

  1. 外部スキーマ

    2. システム利用者から見える機能等を定義するスキーマ。

  2. 概念スキーマ

    3. データベースに保持するデータの要素やデータ同士の関係を記述するスキーマ。

  3. 内部スキーマ

    4. DBMSにどのように格納するか定義するスキーマ。

論理設計

  1. エンティティの抽出

    2. エンティティ (実体)を抽出する。つまり、テーブル名となり得るものを挙げる。

  2. エンティティの定義

    3. エンティティの属性とキー属性を定義する。

    • 主キー (primary key)

      • テーブルに必ず一つ存在し、一行のレコードを一意に識別することができる列の組み合わせ。

    • 外部キー (foreign key)

      • テーブルAとテーブルB間の列同士で設定される。列「col4」がテーブルAにおいて非キーの列であり、 テーブルBにおいて主キーである列であるとする。このとき、テーブルAにおいて列「col4」は外部キーと呼ぶ。

  3. 正規化

    4. エンティティのフォーマットを整理する

  4. ER図の作成

物理設計

  1. テーブル定義
  2. インデックス定義
  3. ハードウェアのサイジング
  4. ストレージの冗長構成決定

    5. 求められる性能と信頼性から、RAIDのレベルとハードディスクの数を決定する

  5. ファイルの物理配置決定

    6. データファイル、インデックスファイル、システムファイル、一時ファイル、ログファイル
